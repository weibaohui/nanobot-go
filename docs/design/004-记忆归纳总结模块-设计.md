## 1. 设计目标

### 1.1 要解决的问题

当前 nanobot 系统缺乏对历史对话的长期记忆能力，每次对话都是独立的，无法记住用户偏好、理解上下文关联、积累知识沉淀。

### 1.2 功能目标

1. **记忆存储与查询**：对外暴露标准接口 `write_memory` 和 `search_memory`
2. **流水记忆层**：短期记忆存储，按日期归类
3. **长期记忆层**：精华抽取后的结构化记忆
4. **记忆升级机制**：定时任务将流水升级为长期记忆
5. **分层检索**：先查流水记忆，再查长期记忆
6. **事件监听**：监听对话完成事件，自动触发记忆写入
7. **模型分离**：支持记忆归纳模型、向量化模型独立配置

## 2. 核心设计思路

### 2.1 架构设计

```
┌─────────────────────────────────────────────────────────────────┐
│                        上层调用方                                │
│         (Agent, API Handler, 定时任务等)                         │
└─────────────────────────────┬───────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│                   MemoryService (统一接口)                       │
│  - WriteMemory(ctx, content, metadata)                          │
│  - SearchMemory(ctx, query, filters)                            │
└─────────────────────────────┬───────────────────────────────────┘
                              │
              ┌───────────────┴───────────────┐
              │                               │
              ▼                               ▼
┌──────────────────────────┐    ┌──────────────────────────┐
│   StreamMemoryRepository │    │ LongTermMemoryRepository │
│   (流水记忆仓储)          │    │ (长期记忆仓储)            │
└───────────┬──────────────┘    └───────────┬──────────────┘
            │                               │
            ▼                               ▼
┌──────────────────────────┐    ┌──────────────────────────┐
│   stream_memories 表     │    │ long_term_memories 表    │
│   (SQLite)               │    │ (SQLite)                 │
└──────────────────────────┘    └──────────────────────────┘
```

### 2.2 模块职责划分

| 模块 | 职责 | 依赖 |
|------|------|------|
| `MemoryService` | 业务逻辑、分层检索、统一接口 | Repository |
| `StreamMemoryRepository` | 流水记忆数据访问 | GORM |
| `LongTermMemoryRepository` | 长期记忆数据访问 | GORM |
| `MemoryEventHandler` | 事件监听、触发记忆写入 | MemoryService, ConversationService |
| `MemorySummarizer` | 初步总结、长期记忆提炼 | LLM Client |
| `MemoryConfig` | 配置管理 | - |

### 2.3 数据流设计

**写入流程：**
```
对话完成
    │
    ▼
事件总线发布 ConversationCompletedEvent
    │
    ▼
MemoryEventHandler 订阅并接收事件
    │
    ▼
根据 trace_id 查询完整对话内容
    │
    ▼
MemorySummarizer 生成初步总结
    │
    ▼
MemoryService.WriteMemory 写入流水记忆
    │
    ▼
返回成功（异步，不阻塞主流程）
```

**查询流程：**
```
调用 SearchMemory(query, filters)
    │
    ├───▶ 查询近期流水记忆（当日/最近N天）
    │
    ├───▶ 查询长期记忆
    │
    ▼
合并结果，按时间+相关性排序
    │
    ▼
返回 SearchResult
```

**定时升级流程：**
```
定时任务触发（每日 05:30-06:30）
    │
    ▼
查询昨日未处理的流水记忆
    │
    ▼
MemorySummarizer 提炼精华
    │
    ▼
生成 LongTermMemory
    │
    ▼
标记流水记忆为已处理
    │
    ▼
（可选）向量化入库
```

```go
// StreamMemory 流水记忆（短期记忆）
type StreamMemory struct {
    ID          uint64    `gorm:"primaryKey" json:"id"`
    TraceID     string    `gorm:"type:text;index:idx_stream_trace_id;not null" json:"trace_id"`
    SessionKey  string    `gorm:"type:text;index:idx_stream_session_key" json:"session_key"`
    ChannelType string    `gorm:"type:text;index:idx_stream_channel" json:"channel_type"`
    Content     string    `gorm:"type:text" json:"content"`           // 原始对话内容摘要
    Summary     string    `gorm:"type:text" json:"summary"`           // AI生成的初步总结
    EventType   string    `gorm:"type:text;index:idx_stream_event_type" json:"event_type"`
    CreatedAt   time.Time `gorm:"type:datetime;index:idx_stream_created_at;not null" json:"created_at"`
    Processed   bool      `gorm:"type:integer;default:0;index:idx_stream_processed" json:"processed"` // 是否已升级为长期记忆
    ProcessedAt *time.Time `gorm:"type:datetime" json:"processed_at,omitempty"`
}

// TableName 指定表名
func (StreamMemory) TableName() string {
    return "stream_memories"
}
```

## 5. 技术决策

### 5.1 为什么选择 SQLite？

1. **零配置**：无需额外部署，适合个人助理场景
2. **单文件存储**：便于备份和迁移
3. **足够性能**：个人助理的数据量，SQLite 完全胜任
4. **与现有系统一致**：对话记录服务已使用 SQLite

### 5.2 模型分离的必要性

| 模型类型 | 用途 | 选择标准 |
|----------|------|----------|
| 主对话模型 | 用户交互 | 高智能、长上下文 |
| 记忆归纳模型 | 总结提炼 | 成本优先、速度优先 |
| 向量化模型 | 语义检索 | 专用 embedding 模型 |

分离的好处：
1. 降低成本（归纳可用轻量级模型）
2. 灵活配置（不同场景选择不同模型）
3. 故障隔离（一个模型不可用不影响其他）

### 5.3 异步写入设计

**原因：**
1. 不阻塞主对话流程
2. 总结需要调用 LLM，耗时较长
3. 失败可重试，不影响用户体验

**实现：**
1. 使用 goroutine 异步执行
2. 记录处理状态，支持幂等
3. 失败时记录日志，可人工介入

### 5.4 分层检索策略

**先查流水记忆的原因：**
1. 近期对话更相关
2. 流水记忆更详细
3. 符合人类记忆习惯（近期更清晰）

**检索顺序：**
1. 当日流水记忆
2. 近期（如最近7天）流水记忆
3. 长期记忆

## 6. 安全考虑

### 6.1 敏感信息过滤

在总结阶段过滤：
- 密码、API Key、Token
- 身份证号、手机号
- 银行卡号

实现：使用正则表达式匹配并替换为 `[FILTERED]`

### 6.2 SQL 注入防护

- GORM 使用参数化查询
- 禁止字符串拼接 SQL

### 6.3 数据隔离

- 预留 multi-tenant 字段（如 user_id）
- 查询时默认按 session_key 隔离

## 7. 性能考虑

### 7.1 索引设计

**stream_memories 表：**
- `idx_stream_trace_id` - 按 trace_id 查询
- `idx_stream_session_key` - 按 session 查询
- `idx_stream_created_at` - 按时间范围查询
- `idx_stream_processed` - 定时任务查询未处理记录
- `idx_stream_channel` - 按渠道查询

**long_term_memories 表：**
- `idx_longterm_date` - 按日期查询

### 7.2 批量处理

- 批量插入使用 `CreateInBatches`
- 定时任务分批处理，避免内存溢出
- 每批默认 100 条

### 7.3 查询优化

- 分页查询限制最大返回数量（100 条）
- 时间范围查询必须指定上限
- 关键词搜索使用前缀索引（如有全文搜索需求，后续迭代）

## 8. 错误处理

### 8.1 错误类型

```go
var (
    ErrMemoryDisabled      = errors.New("memory module is disabled")
    ErrInvalidMetadata     = errors.New("invalid metadata")
    ErrSummarizationFailed = errors.New("summarization failed")
    ErrDuplicateTraceID    = errors.New("trace_id already exists")
    ErrUpgradeFailed       = errors.New("memory upgrade failed")
)
```

### 8.2 错误处理策略

| 场景 | 处理策略 |
|------|----------|
| 写入失败 | 记录日志，返回错误，不阻塞主流程 |
| 总结失败 | 使用原始内容作为总结，记录警告 |
| 定时任务失败 | 记录错误，下次继续处理 |
| 查询失败 | 返回错误，不返回部分结果 |
