## 1. 背景与问题

### 1.1 当前问题

使用 Matrix channel 时，只有大模型完全完成响应后，才会将消息返回给用户。这导致：

1. **用户体验差**：用户需要长时间等待，不知道 AI 正在做什么
2. **缺乏透明度**：用户无法看到 AI 的思考过程、工具调用等中间步骤
3. **其他 channel 也有相同问题**：所有渠道都存在同样的体验问题

### 1.2 期望效果

- 用户能够实时看到 AI 的思考过程
- 工具调用时，用户能够看到正在调用什么工具
- LLM 响应时，用户能够看到助手的回复片段

---

## 2. 功能需求

### 2.1 核心功能

1. **配置开关**：在配置文件中添加开关，控制是否将思考过程发送到 channel
2. **事件监听**：通过观察者模式监听以下事件：
   - 工具调用开始 (`tool_used`)
   - 工具执行完成 (`tool_completed`)
   - 工具执行错误 (`tool_error`)
   - LLM 调用结束 (`llm_call_end`) - 包含助手回复内容
3. **消息发送**：将观察到的事件格式化后发送到对应的 channel

### 2.2 配置设计

```json
{
  "thinkingProcess": {
    "enabled": true,
    "events": ["tool_used", "tool_completed", "llm_call_end"],
    "format": "markdown"
  }
}
```

### 2.3 事件类型

| 事件类型 | 说明 | 发送内容 |
|---------|------|---------|
| `tool_used` | 工具调用开始 | 工具名称、参数 |
| `tool_completed` | 工具执行完成 | 工具名称、执行结果摘要 |
| `tool_error` | 工具执行错误 | 工具名称、错误信息 |
| `llm_call_end` | LLM 调用结束 | 助手回复内容（如果有） |

---

## 3. 非功能需求

1. **性能**：不应显著影响主流程性能
2. **可靠性**：发送失败不应影响主流程
3. **可配置**：用户可以选择启用哪些事件类型

## 5. 验收标准

1. 配置中 `thinkingProcess.enabled = true` 时，能够将事件发送到 channel
2. 配置中 `thinkingProcess.enabled = false` 时，不影响原有行为
3. 可以通过 `events` 配置选择要监听的事件类型
