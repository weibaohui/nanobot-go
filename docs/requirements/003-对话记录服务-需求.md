## 1. 需求背景

### 1.1 现状分析

当前 `nanobot` 项目中，所有对话消息记录存储在 SQLite 数据库的 `events` 表中，由 `agent/hooks/observers/sqlite.go` 负责写入。该实现存在以下问题：

1. **SQL 分散**：SQL 语句直接写在 Observer 中，不利于维护和复用
2. **重复劳动**：其他模块（如 API、统计分析）需要访问对话数据时，需要重复编写 SQL
3. **数据结构不一致**：不同模块可能使用不同的数据结构解析查询结果，导致混乱
4. **难以扩展**：更换数据库或修改表结构时，需要修改多处代码
5. **测试困难**：直接使用 SQL 的代码难以进行单元测试和 mock

### 1.2 需求来源

- 来自项目内部开发需求
- 需要提供统一的数据访问接口供其他模块使用
- 需要支持多种查询场景（按会话、按时间、分页等）

---

## 2. 需求目标

### 2.1 总体目标

创建统一的对话记录 Service 层，封装所有 CRUD 操作，提供标准化的数据访问接口。

### 2.2 具体目标

| 目标项 | 优先级 | 描述 |
|--------|--------|------|
| 统一数据访问封装 | P0 | 封装所有 CRUD 操作，其他模块只调用 Service，不直接写 SQL |
| 按对话ID查询单条原始对话 | P0 | 根据 TraceID 查询完整对话记录 |
| 按会话ID查询对话列表 | P0 | 根据 SessionKey 查询对话列表 |
| 按时间范围查询对话记录 | P0 | 根据时间范围查询对话记录 |
| 分页查询对话列表 | P0 | 所有列表查询支持分页功能 |
| 新增原始对话记录 | P0 | 支持单条和批量新增对话记录 |
| 屏蔽底层数据库细节 | P1 | 表结构、字段、SQL 全部收敛在 Service 内部 |
| 统一返回数据结构 | P1 | 所有调用返回固定结构体，避免各模块解析混乱 |

## 3. 功能需求

### 3.1 核心功能

#### 3.1.1 按对话ID查询单条原始对话

**需求描述**：
- 根据 TraceID 查询该 trace 下的所有消息记录
- 返回按时间正序排列的对话列表

**输入参数**：
- `traceID`: string - 对话追踪ID

**返回结果**：
- `[]ConversationDTO` - 对话消息列表
- 如果不存在，返回空列表

**验收标准**：
- [ ] 能够正确查询指定 TraceID 的所有消息
- [ ] 结果按时间正序排列
- [ ] 返回的数据结构统一使用 ConversationDTO

#### 3.1.2 按会话ID查询对话列表

**需求描述**：
- 根据 SessionKey 查询该会话下的所有消息记录
- 支持分页查询

**输入参数**：
- `sessionKey`: string - 会话标识
- `page`: int - 页码，从 1 开始
- `pageSize`: int - 每页数量

**返回结果**：
- `ConversationListResult` 包含：
  - `conversations`: 对话列表
  - `total`: 总记录数
  - `page`: 当前页码
  - `pageSize`: 每页数量

**验收标准**：
- [ ] 能够正确查询指定 SessionKey 的所有消息
- [ ] 支持分页功能
- [ ] 页码从 1 开始，无效页码自动修正为 1
- [ ] 每页数量限制最大 1000 条

#### 3.1.3 按时间范围查询对话记录

**需求描述**：
- 根据时间范围查询对话记录
- 支持分页查询

**输入参数**：
- `startTime`: time.Time - 开始时间
- `endTime`: time.Time - 结束时间
- `page`: int - 页码
- `pageSize`: int - 每页数量

**返回结果**：
- `ConversationListResult` - 同上

**验收标准**：
- [ ] 能够正确查询指定时间范围内的消息
- [ ] 支持分页功能
- [ ] 时间范围包含边界值

#### 3.1.4 查询最近的对话列表

**需求描述**：
- 获取最近创建的对话记录
- 支持分页查询

**输入参数**：
- `page`: int - 页码
- `pageSize`: int - 每页数量

**返回结果**：
- `ConversationListResult` - 同上

**验收标准**：
- [ ] 返回最近创建的对话记录
- [ ] 支持分页功能
- [ ] 按创建时间倒序查询，返回时按时间正序排列

#### 3.1.5 新增原始对话记录

**需求描述**：
- 创建单条对话记录
- 支持批量创建

**输入参数**：
- 单条：`dto *ConversationDTO`
- 批量：`dtos []ConversationDTO`

**返回结果**：
- 单条：无，成功后 dto.ID 会被填充
- 批量：无

**验收标准**：
- [ ] 能够正确创建单条记录
- [ ] 批量创建使用批量插入，提升性能
- [ ] 空列表不报错

### 3.2 数据结构需求

#### 3.2.1 ConversationDTO

```go
type ConversationDTO struct {
    ID               uint              `json:"id"`
    TraceID          string            `json:"trace_id"`
    SpanID           string            `json:"span_id,omitempty"`
    ParentSpanID     string            `json:"parent_span_id,omitempty"`
    EventType        string            `json:"event_type"`
    Timestamp        time.Time         `json:"timestamp"`
    SessionKey       string            `json:"session_key"`
    Role             string            `json:"role"`
    Content          string            `json:"content"`
    TokenUsage       *TokenUsageDTO    `json:"token_usage,omitempty"`
    CreatedAt        time.Time         `json:"created_at"`
}
```

#### 3.2.2 TokenUsageDTO

```go
type TokenUsageDTO struct {
    PromptTokens     int `json:"prompt_tokens"`
    CompletionTokens int `json:"completion_tokens"`
    TotalTokens      int `json:"total_tokens"`
    ReasoningTokens  int `json:"reasoning_tokens,omitempty"`
    CachedTokens     int `json:"cached_tokens,omitempty"`
}
```

#### 3.2.3 ConversationListResult

```go
type ConversationListResult struct {
    Conversations []ConversationDTO `json:"conversations"`
    Total         int64            `json:"total"`
    Page          int              `json:"page"`
    PageSize      int              `json:"page_size"`
}
```

## 4. 非功能需求

### 4.1 性能需求

| 需求项 | 指标 | 说明 |
|--------|------|------|
| 单条查询响应时间 | < 100ms | 根据 ID 查询单条记录 |
| 列表查询响应时间 | < 500ms | 分页查询 20 条记录 |
| 批量插入性能 | > 1000 条/秒 | 批量插入 100 条记录 |

### 4.2 可靠性需求

| 需求项 | 说明 |
|--------|------|
| 错误处理 | 所有数据库操作必须有错误处理 |
| 数据一致性 | 批量操作使用事务保证一致性 |
| 连接池 | 使用连接池管理数据库连接 |

### 4.3 可维护性需求

| 需求项 | 说明 |
|--------|------|
| 代码分层 | Service → Repository → Database 三层架构 |
| 接口抽象 | 使用接口抽象数据访问层 |
| 测试覆盖 | 核心功能必须有单元测试 |

### 4.4 安全需求

| 需求项 | 说明 |
|--------|------|
| SQL 注入防护 | 使用参数化查询，禁止字符串拼接 |
| 敏感字段保护 | Token 等敏感字段不暴露给外部 |
| 权限控制 | 支持通过 context 传递用户信息进行权限校验 |

## 5. 技术约束

### 5.1 技术栈

- **语言**：Go 1.26+
- **数据库**：SQLite
- **ORM**：GORM
- **测试框架**：testing + testify

### 5.2 依赖约束

- 必须兼容现有代码库
- 不破坏现有功能
- 遵循 `Go-GORM使用规范.md` 规范

### 5.3 兼容性要求

- 保持 SQLiteObserver 功能不变
- 不影响现有 events 表结构
- 向后兼容现有数据
