## 1. 需求背景

### 1.1 现状分析

当前 nanobot 系统缺乏对历史对话的长期记忆能力，每次对话都是独立的，无法：

1. **记住用户偏好**：无法识别用户的长期偏好和习惯
2. **理解上下文关联**：无法关联不同时间的对话内容
3. **积累知识沉淀**：无法从对话中提取有价值的信息长期保存
4. **提供个性化服务**：无法基于历史记忆提供个性化回复

### 1.2 需求来源

- 用户需要 AI 能够"记住"之前的对话内容
- 需要系统自动整理对话精华，形成长期记忆
- 需要支持记忆的分层检索（短期流水记忆 + 长期精华记忆）

---

## 2. 需求目标

### 2.1 总体目标

构建一个独立、可插拔、可扩展的记忆归纳总结模块，实现对话内容的自动采集、流水记录、定时归纳、长期记忆沉淀与统一检索。

模块遵循拟人记忆逻辑：**实时记录 → 睡眠整理 → 分层检索**

### 2.2 具体目标

| 目标项 | 优先级 | 描述 |
|--------|--------|------|
| 记忆存储接口 | P0 | 对外提供标准写入记忆接口 `write_memory(content)` |
| 记忆搜索接口 | P0 | 对外提供标准搜索记忆接口 `search_memory(query, filters)` |
| 流水记忆层 | P0 | 实现短期流水记忆存储（按日期归类） |
| 长期记忆层 | P0 | 实现长期记忆存储（精华抽取后） |
| 记忆升级机制 | P0 | 定时任务将流水记忆升级为长期记忆 |
| 分层检索逻辑 | P0 | 先查流水记忆，再查长期记忆，合并返回 |
| 事件监听触发 | P0 | 监听对话完成事件，自动触发记忆写入 |
| 向量化存储 | P1 | 支持记忆内容的向量化存储与检索 |
| 指代消除 | P1 | 写入流水前进行指代消除，提升记忆质量 |
| 模型分离配置 | P1 | 支持记忆归纳模型、向量化模型独立配置 |

## 3. 功能需求

### 3.1 模块一：记忆存储与查询（核心对外能力）

#### 3.1.1 核心设计原则

- 只对外暴露 **2 个标准接口**：写入记忆、搜索记忆
- 内部自动区分：流水记忆（短期）、长期记忆
- 检索遵循人类习惯：先查近期流水 → 再查长期记忆
- 定时任务自动将流水升级为长期记忆，无需外部干预

#### 3.1.2 写入记忆接口

**接口定义**：`write_memory(content string) error`

**功能描述**：
- 写入流水记忆层（短期记忆）
- 由对话完成后的异步事件模块调用
- 写入内容为对话的初步总结
- 按日期归类到流水账文件夹/表
- 不直接写入长期记忆

**输入参数**：
- `content`: string - 对话初步总结内容
- `metadata`: map[string]interface{} - 元数据（trace_id, session_key, timestamp 等）

**返回结果**：
- `error` - 错误信息

**验收标准**：
- [ ] 能够正确写入流水记忆表/库
- [ ] 按日期自动归类存储
- [ ] 包含完整的元数据信息
- [ ] 写入失败时返回明确错误

#### 3.1.3 搜索记忆接口

**接口定义**：`search_memory(query string, filters SearchFilters) (*SearchResult, error)`

**功能描述**：
- 统一检索所有记忆（内部做分层查询）
- 检索顺序：
  1. 先查当日/近期流水记忆
  2. 再查历史长期记忆
- 返回按时间+相关性合并后的结果
- 对外屏蔽底层结构，上层无需关心存储位置

**输入参数**：
- `query`: string - 搜索关键词/语义
- `filters`: SearchFilters - 过滤条件
  - `startTime`: time.Time - 开始时间
  - `endTime`: time.Time - 结束时间
  - `traceID`: string - 按 trace_id 过滤
  - `sessionKey`: string - 按 session_key 过滤
  - `limit`: int - 返回结果数量限制

**返回结果**：
- `SearchResult` 包含：
  - `streamMemories`: []StreamMemory - 流水记忆结果
  - `longTermMemories`: []LongTermMemory - 长期记忆结果
  - `total`: int - 总结果数
  - `queryTime`: time.Duration - 查询耗时

**验收标准**：
- [ ] 能够正确检索流水记忆层
- [ ] 能够正确检索长期记忆层
- [ ] 结果按时间+相关性排序
- [ ] 支持关键词、时间范围、trace_id 过滤
- [ ] 对外屏蔽底层存储结构

#### 3.1.4 记忆分层与升级规则

**第一层：流水记忆（短期）**

| 属性 | 说明 |
|------|------|
| 来源 | 对话完成 → 初步总结 → 作为一条流水 |
| 特点 | 详细、实时、临时 |
| 生命周期 | 直到被定时任务升级，一直存在 |
| 存储 | 常规数据库 + 可选向量库 |

**进阶要求 - 指代消除**：
- 写入流水前可进行指代消除
- 可读取原始记录，按 trace_id 读取渠道内前后两小时内的对话记录
- 帮助进行指代消除，提升记忆质量

**第二层：长期记忆（深层）**

| 属性 | 说明 |
|------|------|
| 来源 | 每日定时任务 → 对流水记忆做精华抽取 |
| 内容保留 | 发生了什么、结论/结果、价值与用途、高印象事件 |
| 特点 | 精简、结构化、可长期保存 |
| 存储 | 常规数据库 + 向量库 |

**记忆升级机制**：

| 配置项 | 说明 |
|--------|------|
| 触发时间 | 每日固定时间点或时间窗口内执行（可配置） |
| 示例 | 05:30 ~ 06:30 |
| 处理内容 | 读取昨日流水 → 总结 → 生成对应日期的长期记忆 |
| 升级后处理 | 长期记忆生成后自动向量化，存入向量数据库 |

#### 3.1.5 数据库存储设计

**常规数据库**：

| 表名 | 用途 |
|------|------|
| `stream_memories` | 流水记忆表 |
| `long_term_memories` | 长期记忆表 |

**向量数据库**：

| 集合/索引 | 用途 |
|-----------|------|
| `stream_memory_vectors` | 流水记忆向量（可选） |
| `long_term_memory_vectors` | 长期记忆向量（必须） |

**数据模型**：

```go
// StreamMemory 流水记忆
type StreamMemory struct {
    ID          uint64    `gorm:"primaryKey" json:"id"`
    TraceID     string    `gorm:"index" json:"trace_id"`
    SessionKey  string    `gorm:"index" json:"session_key"`
    Content     string    `gorm:"type:text" json:"content"`
    Summary     string    `gorm:"type:text" json:"summary"`  // 初步总结
    EventType   string    `json:"event_type"`  // 对话类型/标签
    CreatedAt   time.Time `gorm:"index" json:"created_at"`
    Processed   bool      `gorm:"default:false" json:"processed"`  // 是否已升级为长期记忆
}

// LongTermMemory 长期记忆
type LongTermMemory struct {
    ID          uint64    `gorm:"primaryKey" json:"id"`
    Date        string    `gorm:"index" json:"date"`  // 日期 YYYY-MM-DD
    Content     string    `gorm:"type:text" json:"content"`  // 精华内容
    WhatHappened string   `gorm:"type:text" json:"what_happened"`  // 发生了什么
    Conclusion  string    `gorm:"type:text" json:"conclusion"`  // 结论/结果
    Value       string    `gorm:"type:text" json:"value"`  // 价值与用途
    Highlights  string    `gorm:"type:text" json:"highlights"`  // 高印象事件（JSON数组）
    VectorID    string    `json:"vector_id,omitempty"`  // 向量库ID
    CreatedAt   time.Time `json:"created_at"`
    UpdatedAt   time.Time `json:"updated_at"`
}

// SearchFilters 搜索过滤条件
type SearchFilters struct {
    StartTime  *time.Time `json:"start_time,omitempty"`
    EndTime    *time.Time `json:"end_time,omitempty"`
    TraceID    string     `json:"trace_id,omitempty"`
    SessionKey string     `json:"session_key,omitempty"`
    Limit      int        `json:"limit,omitempty"`
}

// SearchResult 搜索结果
type SearchResult struct {
    StreamMemories   []StreamMemory   `json:"stream_memories"`
    LongTermMemories []LongTermMemory `json:"long_term_memories"`
    Total            int              `json:"total"`
    QueryTime        time.Duration    `json:"query_time"`
}
```

### 3.2 模块二：事件监听与记忆写入触发

#### 3.2.1 核心职责

- 监听对话最终完成事件
- 自动拉取全量对话 → 将完整对话内容交给记忆模块
- **不做总结、不写流水、不处理归纳**，只做：事件监听 + 内容抽取 + 转发调用

#### 3.2.2 执行流程

```
1. 用户对话 → 生成 trace_id → 全步骤落库
2. 对话执行结束，产生【最终完成事件】
3. 事件处理器捕获事件
4. 根据 trace_id 抽取整轮对话所有内容
5. 将完整对话原文转发给记忆模块，由记忆模块内部完成初步总结
6. 记忆模块完成后，调用 write_memory 写入流水记忆
7. 流程结束，不阻塞主对话
```

#### 3.2.3 技术要求

| 特性 | 说明 |
|------|------|
| 执行方式 | 异步执行，不影响用户对话体验 |
| 幂等性 | 幂等设计，重复事件不重复写入 |
| 容错 | 可配置失败重试策略 |
| 解耦 | 事件监听与记忆处理解耦 |

#### 3.2.4 事件数据结构

```go
// ConversationCompletedEvent 对话完成事件
type ConversationCompletedEvent struct {
    TraceID     string    `json:"trace_id"`
    SessionKey  string    `json:"session_key"`
    ChannelType string    `json:"channel_type"`  // matrix/dingtalk/feishu 等
    StartTime   time.Time `json:"start_time"`
    EndTime     time.Time `json:"end_time"`
    Messages    []Message `json:"messages"`  // 完整对话消息列表
}

// Message 对话消息
type Message struct {
    Role      string    `json:"role"`  // user/assistant/system
    Content   string    `json:"content"`
    Timestamp time.Time `json:"timestamp"`
}
```

### 3.3 模块三：记忆归纳与向量化

#### 3.3.1 初步总结（写入流水前）

**功能**：将完整对话提炼为简短总结

**输入**：ConversationCompletedEvent（完整对话内容）
**输出**：一句话/一段话总结

**提示词要求**：
- 提取对话核心主题
- 保留关键结论
- 去除过程性、重复性内容
- 控制在 200 字以内

#### 3.3.2 长期记忆提炼（每日定时任务）

**功能**：将一日流水记忆提炼为结构化长期记忆

**输入**：当日所有流水记忆
**输出**：LongTermMemory 结构

**提炼维度**：
1. **发生了什么**：概述当日主要对话主题
2. **结论/结果**：重要的决策、答案、解决方案
3. **价值与用途**：这些信息有什么用
4. **高印象事件**：重要、搞笑、特殊的事件（数组）

**提示词要求**：
- 结构化输出
- 信息密度高
- 易于检索匹配

#### 3.3.3 向量化处理

**长期记忆**：
- 生成后必须向量化
- 存入向量数据库
- 用于语义检索加速

**流水记忆**：
- 可选择是否向量化（配置项）
- 建议：高频检索场景可开启

## 4. 模型分离设计（必须独立配置）

系统必须区分**三类不同用途模型**，不共用同一个配置：

### 4.1 主对话模型

| 配置项 | 说明 |
|--------|------|
| 用途 | 用户正常对话、交互、理解意图 |
| 配置路径 | `agents.defaults.model` |
| 特点 | 要求高智能、上下文长 |

### 4.2 记忆归纳模型

| 配置项 | 说明 |
|--------|------|
| 用途 | 对话→流水总结、流水→长期记忆提炼 |
| 配置路径 | `memory.summarization.model` |
| 特点 | 可配置轻量级模型，成本低 |
| 专用提示词 | 支持独立提示词配置 |

### 4.3 向量化模型

| 配置项 | 说明 |
|--------|------|
| 用途 | 记忆内容向量化、检索向量化 |
| 配置路径 | `memory.embedding.model` |
| 特点 | 专用 embedding 模型 |

### 4.4 配置示例

```yaml
memory:
  # 记忆归纳模型配置
  summarization:
    model: "siliconflow/deepseek-v3"
    baseURL: "https://api.siliconflow.cn/v1"
    apiKey: "your-key"
    temperature: 0.3
    maxTokens: 2048
    # 专用提示词
    streamSummaryPrompt: "..."
    longTermSummaryPrompt: "..."

  # 向量化模型配置
  embedding:
    model: "siliconflow/BAAI/bge-large-zh-v1.5"
    baseURL: "https://api.siliconflow.cn/v1"
    apiKey: "your-key"
    dimensions: 1024

  # 定时任务配置
  scheduled:
    upgradeTimeWindow: "05:30-06:30"  # 升级时间窗口
    timezone: "Asia/Shanghai"

  # 存储配置
  storage:
    vectorDB:
      enabled: true
      type: "chromadb"  # 或其他向量库
      path: "~/.nanobot/vector_db"
    streamVectorization: false  # 流水记忆是否向量化
```

### 5.1 模块边界划分

## 6. 扩展能力设计

### 6.1 第三方记忆系统对接

**核心设计**：对外只依赖 `write_memory` + `search_memory`

**替换方式**：
1. 实现 MemoryProvider 接口
2. 替换配置中的 provider
3. 无需改动业务逻辑

```go
// MemoryProvider 记忆提供者接口
type MemoryProvider interface {
    WriteMemory(ctx context.Context, content string, metadata map[string]interface{}) error
    SearchMemory(ctx context.Context, query string, filters SearchFilters) (*SearchResult, error)
    Close() error
}
```

### 6.2 内聚性保证

| 功能 | 归属 |
|------|------|
| 事件监听 | 模块内 |
| 初步总结 | 模块内 |
| 定时归纳 | 模块内 |
| 向量化 | 模块内 |
| 分层检索 | 模块内 |

**目标**：高内聚、低耦合、可插拔、可替换

## 7. 非功能需求

### 7.1 性能需求

| 需求项 | 指标 | 说明 |
|--------|------|------|
| 写入延迟 | < 500ms | 流水记忆写入（异步，不影响主流程） |
| 搜索响应 | < 2s | 分层检索总耗时 |
| 批量升级 | < 10min | 每日流水升级任务 |
| 向量化 | < 1s/条 | 单条记忆向量化 |

### 7.2 可靠性需求

| 需求项 | 说明 |
|--------|------|
| 数据一致性 | 流水记忆升级后标记 processed，避免重复处理 |
| 失败重试 | 写入失败可配置重试策略（次数、间隔） |
| 幂等性 | 相同 trace_id 不重复写入 |
| 降级策略 | 向量库不可用时，降级为文本检索 |

### 7.3 安全需求

| 需求项 | 说明 |
|--------|------|
| 敏感信息过滤 | 总结时过滤密码、Token 等敏感信息 |
| 数据隔离 | 不同用户的记忆数据隔离（多租户预留） |
| 访问控制 | 记忆接口支持权限校验 |
